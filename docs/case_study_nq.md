# Case study: NQ (Natural Questions) — exposure audit

Реальные цифры exposure-аудита на публичном корпусе **BEIR NQ**
(HuggingFace `BeIR/nq`), доказывающие, что метрики retrieval-fairness
работают на масштабе и дают осмысленные числа, а не только на игрушечном
31-чанк демо. Включает regression-diff между lexical (TF-IDF) и dense
(BGE-small) эмбеддерами — killer-feature продукта.

## Условия прогона

- **Корпус:** BEIR NQ, сэмпл 5000 чанков (Wikipedia passages)
- **Запросы:** 500 реальных запросов NQ (test split)
- **Эмбеддеры:** TF-IDF (lexical) и BGE-small-en-v1.5 (dense, fastembed)
- **Стор:** FAISS (IndexFlatIP, cosine на нормализованных)
- **top-k:** 10
- **Время:** TF-IDF ~60 с, BGE ~18 мин (CPU; encode 5500 текстов)

Запуск:
```bash
python scripts/download_nq.py --out data/nq --sample-corpus 5000 --sample-queries 500
python -m retrieval_fairness.case_run --corpus data/nq/corpus.jsonl \
    --queries data/nq/queries.jsonl --embedder tfidf --top-k 10 \
    --out cases/nq_tfidf_sample --html cases/nq_tfidf_sample.html
python -m retrieval_fairness.case_run --corpus data/nq/corpus.jsonl \
    --queries data/nq/queries.jsonl --embedder bge --top-k 10 \
    --out cases/nq_bge_sample --html cases/nq_bge_sample.html
python -m retrieval_fairness diff \
    --baseline cases/nq_tfidf_sample.json --candidate cases/nq_bge_sample.json
```

## Результаты

### Сводные метрики

| Метрика | TF-IDF (lexical) | BGE-small (dense) | Δ (dense − lexical) |
|---|---|---|---|
| Coverage | **52.52%** | **48.00%** | −4.52 п.п. |
| Dark matter | **47.48%** | **52.00%** | +4.52 п.п. |
| Gini | 0.659 | 0.704 | +0.045 |
| Hub capture top5 | 1.9% | 1.8% | −0.02 п.п. |
| Hub capture top10 | 3.3% | 3.4% | +0.01 п.п. |

### Regression-diff (lexical → dense)

| Параметр | Значение |
|---|---|
| Mean per-query overlap | **0.126** (выдачи сильно разные — 87% top-k изменилось) |
| Новых dark-matter (нашлись → не нашлись) | **1155** чанков |
| Спасённых из dark matter (не нашлись → нашлись) | **929** чанков |
| Худшая потеря exposure | doc3969: −24 попаданий |
| Наибольшее улучшение | doc1199: +20 попаданий |

## Что это значит (честная интерпретация)

**Главное число: ~50% корпуса — dark matter при обоих эмбеддерах.**
Половина 5000-чанк корпуса ни разу не попадает в top-10 ни по одному
из 500 реальных запросов. Это реальная, измеримая боль: проиндексировано
5000 чанков, а пользователи через 500 запросов видят только ~2500 из
них. ROI индексации оставшейся половины = 0, и без retrieval-fairness
это невидимо.

**Нетривиальный инсайт regression-diff: переход lexical→dense НЕ улучшает
coverage автоматически.** На этом сэмпле BGE даже немного **ухудшил**
coverage (−4.5 п.п.) и **повысил** Gini (+0.045), но при этом **полностью
перетасовал** top-k (overlap 0.126 — выдачи почти не пересекаются):
929 чанков, которые TF-IDF не находил, BGE находит; 1155 чанков, которые
находил TF-IDF, BGE теряет. То есть dense не «строго лучше», а «другой» —
и какая половина корпуса останется dark matter, **зависит от выбора
эмбеддера**. Это ровно та боль «миграция вслепую», ради которой сделан
regression-diff: без него команда сменила бы TF-IDF на BGE и не узнала,
что 1155 релевантных чанков перестали находиться.

**Hub capture умеренный** (top10 ≈ 3% у обоих): «хабового захвата»
на NQ нет — 500 запросов достаточно разнообразны, чтобы не стекаться
к 10 чанкам. Это отличает NQ от синтетических корпусов с дубликатами,
где hub capture может быть экстремальным (см. smoke-тест в коммите
`9ce6d37`).

## Сравнение с T-Retrievability (MS MARCO) — честно

T-Retrievability (источник наших метрик) публиковала retrievability на
**MS MARCO dev set**. Прямое числовое сравнение **некорректно** без
выравнивания:
- другой корпус (MS MARCO vs NQ);
- другой query workload (разный размер, разное распределение запросов);
- другой top-k / normalization.

Что можно сказать честно: наш Gini 0.66–0.70 и dark-matter ~50% —
попадают в **ожидаемый диапазон** для retrieval-систем на публичных
корпусах (не экстремум). Это делает кейс **репрезентативным**, а не
нагнетающим. Прямое числовое сопоставление с T-Retrievability оставлено
как TODO (Шаг 9.6 опционально) — требует того же эмбеддера/корпуса/top-k.

## Killer-feature в действии

Regression-diff lexical→dense — это та продуктовая обёртка, которой
голые скрипты T-Retrievability не дают. T-Retrievability считает Gini
на одном прогоне; мы показываем **что именно поменяется при миграции
эмбеддера**: какие чанки станут dark matter, какие спасутся, насколько
изменится выдача по запросам. Команда, мигрирующая TF-IDF→BGE, запускает
одну команду и видит: «929 чанков оживут, 1155 умрут, выдача по 87%
запросов изменится» — и принимает решение осознанно, а не вслепую.

## Валидация через qrels (дополнительный инсайт)

BEIR NQ содержит qrels (оценки релевантности запрос→чанк).
`scripts/qrels_validate.py` отвечает: **сколько dark-matter чанков на
самом деле релевантны каким-то запросам по qrels?**

```bash
python scripts/qrels_validate.py --probe cases/nq_tfidf_sample.json \
    --qrels data/nq/qrels.json --queries data/nq/queries.jsonl
```

| Метрика | TF-IDF | BGE-small |
|---|---|---|
| Релевантных чанков (qrels) в сэмпле | 158 | 158 |
| «Потерянное золото» (dark И релевантные) | **34** (21.5% релевантных) | **8** (5.1% релевантных) |
| Recall@10 по qrels | **62.7%** (99/158 пар) | **91.1%** (144/158 пар) |

**Инсайт: coverage и релевантность — разные оси.** BGE *хуже* по
coverage (−4.5 п.п., dark matter больше на 226 чанков), но *сильно лучше*
по релевантности: recall@10 91% против 63%, а «потерянного золота»
вчетверо меньше (8 vs 34). То есть dense-эмбеддер жертвует шумовой
частью корпуса (его dark matter — на 99.7% нерелевантные чанки),
но надёжнее достаёт релевантное. Это подтверждает продуктовый тезис:
exposure-метрики и qrels-метрики **дополняют**, а не заменяют друг
друга — coverage ловит «мертвый груз индекса», qrels-валидация
показывает, болит ли это для качества ответов.

Оговорка: qrels покрывают только 158 из 5000 чанков сэмпла (3%) —
остальной dark matter «не оценён», а не «доказанно нерелевантен».

## Вывод по кейсу

Кейс отвечает на главный вопрос дорожной карты: **«даёт ли
retrieval-fairness осмысленные цифры на реальном масштабе?»** — да:
на 5000-чанк сэмпле NQ с 500 реальными запросами инструмент за ~60 с
(TF-IDF) / ~18 мин (BGE) выдал coverage ~50%, dark-matter ~50%, Gini
0.66–0.70, и regression-diff показал, что смена эмбеддера перетасовывает
dark matter (929 спасённых / 1155 потерянных) при overlap 0.126. Это
реальная, измеримая, нетривиальная картина, невидимая без аудита
exposure.

## Честные оговорки

- **Сэмпл 5000/500, не полный NQ (260k).** Полный прогон требует ресурсов
  и времени (BGE на CPU: ~18 мин на 5k → ~15 ч на 260k). Для полного
  прогона есть GPU-пайплайн: тяжёлый encode в Colab
  (`scripts/colab_nq_bge_encode.py`, ~30–60 мин на T4), лёгкий probe
  локально (`scripts/probe_precomputed.py`, минуты, влезает в 16 GB RAM).
  Цифры на сэмпле репрезентативны по порядку величины, но не
  точные значения полного корпуса.
- **TF-IDF на больших корпусах требует `--max-features`** (например
  4096): без ограничения словарь на 260k пассажей — сотни тысяч
  измерений, и денсификация под FAISS не влезает в RAM (case_run
  и TfidfEmbedder теперь предупреждают об этом).
  Оговорка сравнимости: сэмпл-прогон 5k выше сделан без ограничения
  словаря — tfidf-цифры полного прогона с max_features будут из чуть
  другого пространства признаков.
- **BGE на CPU медленный** (18 мин на 5500 текстов). На GPU или с
  батчингом — быстрее; это ограничение окружения, не продукта.
- **Прямое сравнение с T-Retrievability** требует выравнивания
  эмбеддера/корпуса/top-k — TODO (Шаг 9.6).
- **qrels-валидация на сэмпле покрывает лишь 3% чанков** — вывод про
  «шумовой» dark matter качественный, не количественный.
- **TF-IDF и BGE — не все варианты.** Другие эмбеддеры (E5, BGE-large,
  OpenAI) дадут иные цифры; кейс демонстрирует метод, не исчерпывает
  пространство.
